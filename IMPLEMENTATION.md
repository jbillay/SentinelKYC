## Phase 2 — Screening Agent

Big new step in the POC: a screening sub-graph that runs after `synthesize_card` and before `done`. Screens compiled subjects (company + officers + PSCs + extracted shareholders) against locally cached sanctions lists (OFAC SDN + UK HMT) and live adverse-media (Bing News). Each potential hit is evaluated by the reasoning LLM (true match / false positive / needs review), captured as a nested decision fragment under its parent `evaluate_*` fragment, and persisted to a new `screening_hits` / `screening_evaluations` pair of tables. A new Screening tab on the dossier lets the human override any LLM decision.

**Single source of truth for the design**: `SCREENING_PLAN.md` — covers data model, graph wiring, prompts, HTTP API, UI surfaces, milestones, locked-in assumptions, and explicit out-of-scope. **Read it before starting any milestone below.** Tasks here are the execution checklist; design changes go in the plan first.

PEP screening, recursive ownership-chain walking, authorized signatories, real-time list refresh, alias generation, and screening-in-the-run-diff are explicitly **out of scope for v1** (see §11 of the plan).

# Phase 3 — Risk Assessment Engine

Deterministic, rules-based, weighted-factor risk engine enriched by an LLM rationale, slotting in **after** `compile_screening_report` in both `compiledGraph` and `compiledScreeningOnlyGraph`. Reads `kycCard` + `profile` + `psc` + `screeningReport.summary`; emits a versioned calculation receipt + score + tier (Low / Medium / High) + outcome (Low / Medium / High / Prohibited) + delta vs. previous run + LLM rationale, persisted to `runs.final_risk_assessment`. The configurable matrix lives in two new Postgres tables (`risk_matrix_versions`, `risk_matrix_active`) and is edited via API + Settings page.

**Locked-in design decisions** (see clarification round on 2026-05-10):

- **v1 factors (lean)**: Geographic (entity country), Entity (type + structural complexity from PSC layers + corporate-PSC count), Industry (SIC code prefix). UBO nationality / channel / financials deferred.
- **Config storage**: Postgres tables, mirroring the prompt registry (versioned, append-only, singleton-active).
- **LLM role**: Rationale generation (always) + free-text country → ISO-2 normalization (deterministic lookup table first; LLM only on miss; result cached in `kv_cache`).
- **Knockouts**: Read `screeningReport.summary` as single source of truth — confirmed sanctions → `Prohibited`; `overallRisk === 'high'` → force `High`; `overallRisk === 'medium'` → floor at `Medium`. PEP knockout deferred (no upstream PEP data in v1).
- **Trajectory**: Compute Δ vs. previous run's persisted score; flag in UI if `|Δ| ≥ 15` (configurable on the matrix).
- **Recompute paths**: Runs in both `compiledGraph` and `compiledScreeningOnlyGraph` (so `refresh` and `rescreen` both refresh the score). Plus a new cheap `POST /api/dossiers/:cn/recalculate-risk` for "matrix-edit-only" rebases.
- **Knockout triggers are enumerated in code, not eval'd from the JSON** — declarative tag → handler in `services/risk/knockouts.js`. Avoids shipping an expression engine in a POC.

Six-milestone delivery (R1 → R6).

# Phase 4 — Pluggable LLM backend (Ollama ↔ NVIDIA NIM)

Make the LLM backend swappable **per task** (`ocr`, `reasoning`) behind a small provider abstraction, so a run can use local Ollama, NVIDIA's hosted NIM API, or (later) a third backend — selected by env var. v1 ships two providers: **`ollama`** (today's behaviour, unchanged) and **`nvidia`** (NVIDIA-hosted API at `build.nvidia.com` — no container, just an `nvapi-…` key). Replaces `glm-ocr` with the dedicated **NeMo Retriever OCR v2** NIM (`nvidia/nemoretriever-ocr-v2`) and `llama3.1:8b` with **`meta/llama-3.1-8b-instruct`** on NIM. The shared layer keeps everything that isn't provider-specific: the `extract.json_strict_retry` retry wrapper, the OCR file-read + `kv_cache` cache, `forceFresh` plumbing, the prompt registry, the Zod schemas, rasterization, the OCR page cap. Health surfaces the actual provider + model in use **per task**.

**Locked-in design decisions** (clarification round 2026-05-11):

- **Hosting**: NVIDIA-hosted API only — `https://integrate.api.nvidia.com/v1` for chat, the NeMo OCR CV endpoint (`/v1/infer`) for OCR. No NIM container anywhere — Docker is forbidden on this machine.
- **Granularity**: per-task selection — `LLM_OCR_PROVIDER` and `LLM_REASONING_PROVIDER` independently, each defaulting to `ollama` (an unconfigured `.env` behaves exactly as today). Optional `LLM_PROVIDER` convenience var sets both unless a per-task var overrides.
- **Config mechanism**: env vars + server restart (POC — no Settings-page toggle, no DB table). A runtime/Settings provider selector is an explicit non-goal for v1.
- **NVIDIA OCR model**: `nvidia/nemoretriever-ocr-v2` (a dedicated OCR NIM, not a VLM). It is **not** chat-completions — it's a `POST …/v1/infer` endpoint taking base64 `image_url` objects (`{ "input": [{ "type": "image_url", "url": "data:image/png;base64,…" }], "merge_levels": [...] }`) and returning text detections + bounding boxes + confidences. The NVIDIA OCR provider flattens that into the reading-order text string the extractors already expect — so nothing downstream of `ocrPage()` changes.
- **NVIDIA reasoning model**: `meta/llama-3.1-8b-instruct` — the literal equivalent of the current `llama3.1:8b` (minimal prompt drift). Driven via `@langchain/openai`'s `ChatOpenAI` against `configuration.baseURL = https://integrate.api.nvidia.com/v1`; `withStructuredOutput` with a method knob (`functionCalling` default; `jsonMode` fallback if NIM tool-calling misbehaves). Note the langchainjs quirk: the base URL must be passed as nested `configuration.baseURL`, not a top-level `baseURL`/`basePath`.
- **No cross-provider fallback** — if the configured provider is unreachable the run fails loudly (same as Ollama being down today). Fallback chains are out of scope.
- **OCR cache key gains provider+model** — `kv_cache` ns `ocr` key changes from `sha256(image)` to `${sha256(image)}:${provider}:${model}`, so switching OCR backends can't serve the other model's cached text. Old bare-hash entries simply stop being hit (harmless).
- **`services/ollama.js` is replaced, not shimmed** — new `services/llm/` module; the ~6 import sites move; no back-compat re-export (per the repo's own guidance).

Five-milestone delivery (P1 → P5).

# Phase 5 — QA Agent + Human Decision Panel

Two coupled additions: (1) a **QA AI Agent** that runs after `assess_risk` in both `compiledGraph` and `compiledScreeningOnlyGraph`, validates the just-built case (completeness + cross-agent consistency), and routes it into one of `auto_approved` / `streamlined_review` / `standard_review`; (2) a **Final Decision Panel** on `DossierViewPage` for the reviewer to Approve / Reject / Escalate / Request Info, persisting the action to an immutable audit fragment and redirecting to `/dossiers`.

**Locked-in design decisions** (see clarification round on 2026-05-14):

- **QA placement**: new `qa_check` graph node after `assess_risk`, wired into **both** compiled graphs (refresh + rescreen both refresh QA). Plus `POST /api/dossiers/:cn/runs/:runId/qa/recompute` for matrix-threshold-edit-only rebases (mirrors `recalculate-risk`). No new graph thread.
